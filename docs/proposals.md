# 新功能提案

**Version**: 1.1 | **Date**: 2026-04-29

本文档收录所有待实现的功能提案与增强建议，按优先级分类。每条提案包含目标、设计思路、工时预估及依赖关系。

---

## 高优先级

### FEAT-006: 单元测试覆盖

- **目标**: 为核心算法模块建立单元测试，保障重构和迭代的安全性
- **设计方案**: 分两期推进
  - **P0（优先）**: SequenceAligner, Slicer, AlignmentDecoder。这三个模块逻辑复杂、输入输出明确，最适合也最需要测试覆盖
  - **P1（其次）**: DictionaryG2P, ChineseProcessor
- **预估工时**: P0 约 16h，P1 约 8h
- **依赖**: 需确认测试框架选型（建议 QTest 或 GoogleTest）

## 中优先级

### FEAT-002: 批量重采样工具

- **目标**: 提供批量音频重采样功能，支持常见采样率转换
- **设计方案**: 复用现有 TaskWindow 异步任务框架 + audio-util 库的重采样能力，新增一个独立的工具窗口
- **预估工时**: 4h
- **依赖**: dstools-audio, audio-util

### FEAT-003: AudioSlicer 参数校验

- **目标**: 防止用户输入不合法的切片参数导致异常行为
- **设计方案**: 在 UI 层和逻辑层分别添加校验规则：
  - minInterval <= minLength
  - hopSize <= minInterval
  - threshold < 0
- **预估工时**: 1h
- **依赖**: 无

### FEAT-004: LyricFA 线程安全

- **目标**: 解决 Asr::recognize 在多线程调用下的数据竞争问题
- **设计方案**: 为 Asr::recognize 方法添加 std::mutex 保护。考虑到推理本身较重，互斥锁的开销可以忽略
- **预估工时**: 1h
- **依赖**: FunAsr 推理库

---

## 低优先级

### FEAT-007: 统一配置系统 ⚠️ 部分完成

- **目标**: 统一 5 个 EXE 的配置管理方式，消除 key 命名不一致的问题
- **剩余工作**: 3 个应用仍使用旧的 INI 配置，需迁移到 AppSettings 并统一 key 命名规范
- **预估工时**: 4h（迁移 + 验证）
- **依赖**: 无

### FEAT-005: 跨平台 CI/CD ⚠️ 部分完成

- **目标**: 在 GitHub Actions 上建立全平台持续集成流水线
- **剩余工作**: macOS 和 Linux 的 CI 流水线（Windows Release 构建已完成）
- **设计方案**: 以 Windows workflow 为参考模板，处理各平台的依赖安装、编译、打包
- **预估工时**: 6h（仅 macOS + Linux）
- **依赖**: FEAT-006（有测试覆盖后 CI 才有实质意义）

### FEAT-008: 结构化日志

- **目标**: 替换零散的 qDebug 输出，引入分类日志便于问题排查
- **设计方案**: 使用 QLoggingCategory 为各模块定义日志类别，支持按类别过滤和输出级别控制
- **预估工时**: 4h
- **依赖**: 无

### 建议推进顺序

1. **FEAT-003**（参数校验）+ **FEAT-004**（线程安全）：两者相互独立，可并行推进
2. **FEAT-006 P0**（核心测试）：为后续重构提供安全网
3. **FEAT-002**（批量重采样）：独立功能，随时可做
4. **FEAT-005**（CI/CD 剩余平台）：等 FEAT-006 有一定覆盖后搭建 macOS/Linux
5. **FEAT-007**（配置迁移）+ **FEAT-008**（结构化日志）：低风险，可穿插进行
6. UI/视觉体系的各项工作，等核心功能和测试稳定后再集中推进
