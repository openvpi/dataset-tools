## 关键发现

1. **无循环依赖** - 依赖流向清晰, 分层合理
2. **两套音频系统并存** - audio-util (C++ 原生) 与 qsmedia (Qt NAudio 移植) 功能重叠但互不依赖
3. **推理模块高度重复** - some-infer / rmvpe-infer / game-infer 共享约 300+ 行相同代码, 缺少公共基础库
4. **FunAsr 完全独立** - 与其他模块无代码共享, 代码风格为 C 风格, 质量最低

## 建议的目标架构

```
  some-infer ─┐
  rmvpe-infer ┼──► infer-common (新建)──► audio-util
  game-infer ─┘         │                     │
                        ▼                     ▼
                   onnxruntime          libsndfile/soxr/mpg123
```

提取 `infer-common` 后, 三个推理模块的重复代码可减少 80% 以上。

## 深层架构问题

### 1. 所有权语义缺失
整个代码库缺乏清晰的所有权语义:
- FunAsr 使用原始指针传递数据, 无法判断谁负责释放
- SpeechWrap 持有外部传入的裸指针, 无生命周期保证
- FeatureQueue 存储 `new Tensor*`, 但消费者可能不 pop
- Audio::frame_queue 同样存储裸指针且析构不清理

**建议**: 建立统一的所有权约定 - 生产者 `std::unique_ptr` 转移, 消费者 `std::move` 接收。

### 2. 错误处理策略不一致

| 模块 | 错误处理方式 |
|------|-------------|
| audio-util | `std::cout` 打印 + 返回空对象 |
| qsmedia | Qt 信号 / 返回 bool |
| ffmpegdecoder | `goto` 跳转 + 返回 false |
| FunAsr | 完全无错误处理 (崩溃) |
| some-infer | `std::cerr` 打印 + 返回空 vector |
| game-infer | 同 some-infer |

**建议**: 统一为 Qt 风格 - 关键错误用 `qCritical()`, 一般错误返回 `std::optional` 或 `expected`, 致命错误抛异常。

### 3. 线程模型混乱

- sdlplayback: SDL 回调线程 + 主线程, 用 volatile 同步 (错误)
- ffmpegdecoder: 单线程但控制流复杂
- FunAsr: 无线程考虑但 FeatureQueue 暗示多线程使用
- 推理模块: 单线程使用但 static 变量不安全

**建议**: 为每个模块明确文档化线程模型 (单线程 / 线程安全 / 回调线程), 并用适当的同步原语实现。

### 4. VIO 抽象层缺陷

`SndfileVio` 作为内存中虚拟 I/O 层, 存在三个逻辑 bug (seek 不检查边界, read 无符号下溢, write 不更新 seek)。这意味着任何通过 VIO 处理音频的路径都可能产生静默的数据错误。

**建议**: 为 SndfileVio 添加完整的单元测试, 覆盖 seek 各模式、read 越界、write+read 交替等场景。

### 5. FunAsr 技术债评估

FunAsr 模块占全部问题的 36% (19/53), 且包含 4 个 CRITICAL 级别问题。该模块是一个 C 语言库的直接移植, 几乎没有 C++ 现代化。

| 指标 | 状态 |
|------|------|
| RAII | 无 - 全部手动内存管理 |
| 异常安全 | 无 - 任何异常都会泄漏 |
| 边界检查 | 无 - 直接索引无验证 |
| NULL 检查 | 无 - malloc/fopen 不检查 |
| 线程安全 | 不适用 - 无同步设施 |
| 现代 C++ | 无 - C89 风格代码 |

**建议**: 考虑是否值得维护。如果 FunAsr 功能可被其他 ASR 库替代 (如 whisper.cpp), 建议评估替换成本。否则需要 5-8 天专项治理。
